# Legacy Auth Flow Review

This note records the current login, registration, and session behaviour before the clean database path creates any fresh local accounts.

The goal is not to rewrite auth in this pass. The goal is to stop us from accidentally generating account fixtures that do not match the current runtime.

## Files inspected

```text
game-full/index.php
game-full/login/login.php
game-full/register/index.php
game-full/register/create-new-weevil.php
game-full/essential/backbone.php
game-full/essential/config.php
game-full/essential/internal.php
tools/create_local_account.py
```

## Current login flow

1. `game-full/index.php` renders the returning player form.
2. The form posts `userID` and `password` to `game-full/login/login.php`.
3. `verifyUser()` looks up the user by `username`.
4. The submitted password is compared directly with `users.password`.
5. If the account is active and not banned, the runtime generates a new `sessionKey` and `loginKey`.
6. The `users` row is updated with the new keys, `lastLogin`, and `loginIP`.
7. The browser receives `sessionId` and `weevil_name` cookies.
8. The player is redirected to `game-full/game.php`.

## Current registration flow

1. `game-full/register/index.php` renders the registration page.
2. The page posts `userID`, `password`, and `recap` to `game-full/register/create-new-weevil.php` with XHR.
3. `createWeevil()` inserts a row into `users`.
4. The submitted password is inserted directly into `users.password`.
5. The new account receives a generated `sessionKey` and `loginKey` immediately.
6. The helper creates a buddy list for the new username.
7. The same `sessionId` and `weevil_name` cookies are set and the player is redirected to `game-full/game.php`.

## Current session bridge

The Flash/PHP compatibility layer still depends on the legacy `sessionKey` and `loginKey` columns.

`confirmSessionKey()` checks the cookie username and `sessionId` against the `users` table. `getLoginDetails()` returns the `loginKey`, user id, username, and tycoon state in the old query-string style expected by the game runtime.

That means any local account generator must create compatible `users.sessionKey` and `users.loginKey` values until the runtime is migrated away from that legacy bridge.

## Findings

- Passwords are currently stored and checked as plain text.
- Admin login helpers also check `users.password` directly.
- `generateSessionKey()` and `generateLogKey()` use `rand()`, which is compatible with the old code but not modern session security.
- Login cookies are set without `HttpOnly`, `Secure`, or `SameSite` options.
- Registration defines reserved-name/profanity helpers, but the current account creation path does not call all of that validation before inserting the row.
- The registration page had a hardcoded reCAPTCHA secret value even though the current script does not actually perform server-side verification.
- The clean seed path correctly blocks legacy `users`, `buddylist`, `buddyalerts`, `nest`, `nestinfo`, and `weevilitems` INSERT groups from default catalogue seeds.

## Compatibility rule for the clean database path

Do not import old account rows from `bwps.sql` into the default clean path.

Fresh local accounts should be generated by tooling only, with names that are obviously local fixtures, for example:

```text
local_admin
local_demo
```

Those accounts should be safe for local testing and should not pretend to be old moderators, celebrities, staff, or real player accounts.

## Password decision before account writes

There are two possible next steps:

1. Keep the first local account writer legacy-compatible and clearly local-only. This means writing the password in the current plain-text format because the old login code expects it.
2. Modernise the PHP login/register helpers first so they can use `password_hash()` and `password_verify()`, while still allowing existing local legacy rows during migration.

The safer long-term route is option 2. The faster compatibility route is option 1, but it must remain local-only and must be labelled as unsafe for production.

## Recommended next pass

- Add a small PHP password compatibility helper.
- Update login/admin auth checks to use the helper instead of direct string comparison.
- Update registration to write `password_hash()` values for newly created accounts.
- Then let `tools/create_local_account.py` perform real local account writes using hashed passwords.

Until that pass lands, `tools/create_local_account.py --execute` should continue refusing to write accounts.
